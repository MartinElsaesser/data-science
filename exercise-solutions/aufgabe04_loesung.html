def confusion_matrix ( y, pred ):
    TP = (y==1) & (pred == 1)
    FP = (y==0) & (pred == 1)
    TN = (y==0) & (pred == 0)
    FN = (y==1) & (pred == 0)

    TP = TP.sum()
    FP = FP.sum()
    TN = TN.sum()
    FN = FN.sum()
    
    return {"TP":TP,"FP":FP,"TN":TN,"FN":FN}

def calculate_metrics ( TP, FP, TN, FN ):
    # accuracy
    acc = (TP+TN)/(TP+TN+FP+FN) if TP+TN+FP+FN>0 else np.nan
    
    # sensitivity
    sensitivity = TP/(TP+FN) if TP+FN>0 else np.nan
    
    # specificity
    specificity = TN/(TN+FP) if TN+FP>0 else np.nan
    
    # precision
    precision = TP/(TP+FP) if TP+FP>0 else np.nan
    
    # recall = sensitivity
    recall = TP/(TP+FN) if TP+FN>0 else np.nan
    
    return {"accuracy": acc,
            "sensitivity": sensitivity,
            "specificity": specificity,
            "precision": precision,
            "recall": recall}

Teilaufgabe 4¶

Precision-Recall-Tradeoff: Binäre Klassifier liefern einen Score (via predict_proba), der eine Zahl zwischen 0 und 1 ist (oder eine Zahl zwischen −∞−∞ und +∞+∞ via predict_log_proba). Diese Zahl wird dann via einem Threshold in die Klasse 0 oder 1 übersetzt.

Wenn der Threshold sehr niedrig gesetzt wird, dann werden alle oder fast alle Samples als 1=krank klassifiziert. Dies liefert eine hohe Sensitivität/Recall (alle oder fast alle, die positiv sind, werden als positiv klassifiziert), aber eine niedrige Precision (von den positiv Klassifizierten sind nur wenige tatsächlich positiv).

Wenn der Threshold sehr hoch gesetzt wird, dann werden alle oder fast alle Samples als 0=gesund klassifiziert. Dies liefert eine niedrige Sensitivität/Recall (wenige, die positiv sind, werden als positiv klassifiziert), aber eine hohe Precision (von den positiv Klassifizierten (diese sind die eindeutigen Fälle mit einem hohen Score) sind die meisten tatsächlich positiv).

Man kann via Wahl des Thresholds nicht gleichzeitig Precision und Recall maximieren. Daher findet ein Tradeoff statt, der je nach angestrebtem Use Case optimiert wird (unter Berücksichtigung der Kosten von False Negatives und False Positives).

Tutorial: Im Tutorial muss der Threshold etwa auf 0.52 eingestellt werden.

def draw_rect ( ax, x,y, decision, gini, samples, value, width=0.95, height=0.3, has_children=False ):
    if has_children:
        dx=0.5
        dy=0.3
        ax.arrow ( x=x, y=y, dx=-dx, dy=-dy, color="black", width=0.01 )
        ax.text  ( s="True", x=x-dx*0.8+0.1, y=y-dy*0.8 )
        ax.arrow ( x=x, y=y, dx=dx, dy=-dy, color="black", width=0.01 )
        ax.text  ( s="False", x=x+dx*0.8+0.1, y=y-dy*0.8 )
    
    ax.add_patch ( patches.Rectangle(xy=(x-width/2,y-height/2),
                                     width=width, height=height, facecolor="white",edgecolor="black" ))
    ax.text ( s="\n".join ( [decision,
                             "gini={0:.3f}".format(gini),
                             "samples={0}".format(samples),
                             "value={0}".format(value)] ),
              x=x, y=y, ha="center", va="center" )

Diskussion: Der letzte Split in Cases 1A, 1B ist nicht ganz falsch -- er verringert den Gini-Wert von 0.33 auf 0.31. Allerdings würde die Klassifikation auch im Fall 1B (unterste Zeile, rechts) "krank" liefern, denn in dieser Menge sind 31 gesunde, aber 39 krank. entgegen unserem Algorithmus (hier haben wir "gesund" gesagt).

In der Vorlesung haben wir ("ein einfacher Baum" in Abschnitt 3.3) ähnliches gesehen: Der linke Teilbaum hat nur Blätter, die als Ergebnis "krank" liefern.

Warum splitten wir überhaupt nochmal?

  1. Wir könnten den Baum tiefer machen und diesen Knoten als Ausgangspunkt eines weiteren Splits machen.
  2. Auch ohne weiteres Splitting liefert dieser Split "Sex=='M'" eine Zusatzinformation: via clf.predict_proba wird nicht die vorhergesagte Klasse ermittelt (diese wäre 1=krank im Fall 1B), sondern die Proportion der Kranken: 39/(31+39). Im Fall 1B wüssten wir also -- im Vergleich zum Fall 1A -- dass die Klassifikation mit einer erheblich größeren Unsicherheit behaftet ist.
